# Business Checkup — Build Brief

## What we're building
An automated "Business Checkup" audit tool bolted onto the existing goldentoolbox.com static site. A visitor submits a short form (name, business, email, phone, website — no address), the system resolves their business via Google Places, runs a multi-source audit pipeline, and produces a thorough, free, shareable report page at `/checkup/[slug]` scoring 8 categories into one overall score, each tied back to Golden Toolbox's Four Tools pitch.

## For whom / why
- **Primary audience:** local trades / home-service owners (concrete, plumbing, HVAC, electrical, roofing, etc.) — the same audience as the marketing site.
- **Secondary:** Jared, live on sales calls, via an authenticated admin trigger that skips the public form.
- **Outcome:** a concrete, personalized "here's what you're missing" artifact that makes the free-checkup CTA tangible and drives leads.

## In scope (this version)
- Form flow: collect fields → **business confirmation step** (top 1–3 Places matches when no confident auto-match) → cached-report check → generate → redirect to report.
- **Async generation** with a loading state (PageSpeed can take 10–20s+).
- Report page `/checkup/[slug]`: fully thorough (no teaser tier), 8 categories + rolled overall score, Four-Tools framing on low scores, site-styled CTA at the end. Shareable link + print/PDF export. Mobile-first.
- 8 scoring categories: Business Details, Techno Stack, Google Business Profile, Directory Presence (Google/Yelp/Facebook), Online Reputation, Website Performance, Speed-to-Lead, Local Competitor Snapshot.
- **Lead gating:** generation only after form submit; no public unauthenticated generation endpoint.
- **Cost control:** 30-day cache keyed on domain/Place ID; rate limit (max 3 new generations per IP/email per hour); per-generation spend log of which API calls fired.
- Admin-only manual trigger (authenticated) for sales calls.

## Out of scope (for now)
- Paid review aggregators (structure Reputation so one can be swapped in later for full history).
- Carrier-verified phone lookup (Speed-to-Lead is pattern-based "estimated" via libphonenumber-js).
- Full historical reviews beyond the ~5 the Places API returns.
- Any redesign of the existing marketing page.

## Design direction
Reuse the existing design system verbatim: `--paper #F6EFE3`, `--ink #221B12`, gold ramp (`--gold #C9962B` / `--gold-hi` / `--gold-lo`), Playfair Display (display) + Manrope (body), existing `.btn`/`.btn-gold`/`.eyebrow`/`.chip`/`.phone` component vocabulary, `.reveal` scroll animations, mobile-first breakpoints (1150/860/760). Report visuals (score rings, category bars, competitor leaderboard) styled from the same tokens. End with the site's "Get my free Business Checkup" CTA styling.

## Technical decisions (confirmed)
- **Architecture:** keep the static site exactly as-is; add Vercel serverless functions in `/api` + new pages for the form flow and report. No framework migration.
- **Storage:** Upstash Redis free tier (no card) — cached reports w/ 30-day TTL, rate-limit counters, spend logs.
- **Places API:** enable Google Maps Platform billing and live within the free monthly credit; hard usage caps + spend logging so cost stays effectively $0. Built behind a clean adapter so the whole flow runs on **mock data first** (priority #1), then flips to real.
- **PageSpeed Insights:** genuinely free (API key only, no billing).
- **Phone type:** libphonenumber-js, offline, labeled "estimated."
- **Lead notify:** keep firing Formspree (so the lead still hits Jared's inbox) in addition to generating the report.

## Build order (confirmed priority)
1. Full flow end-to-end on **mock data** (form → confirm → cache-check → report UI).
2. Real PageSpeed integration.
3. Real techstack + chat-widget HTML scan.
4. Real Google Places (GBP, reviews, competitor, address resolution).
5. libphonenumber-js phone estimate.
6. Directory presence checks (Google/Yelp/Facebook).
7. Caching + rate limiting (real Upstash).
8. Admin-only manual trigger route.

## Post-launch fixes (after real usage surfaced real bugs)
- **Fabricated reply-rate stat removed.** The user caught this live: Online Reputation showed "79% reply rate" while every visible review sample read "No reply" — a real credibility bug, not a display glitch. Root cause: neither Google Places nor an HTML scan can see review-reply data, so a leftover mock placeholder number was silently carried through into live reports. Fixed in both `places.js` (Online Reputation) and `htmlscan.js` (Business Details) to honestly show "Not available" and rescaled both scoring formulas to not credit/penalize something never measured. Individual review samples now omit the reply line entirely when unknown instead of asserting "No reply". Verified against the exact live business from the bug report.
- **Competitor ranking formula fixed.** User caught a 3.3★/176-review business ranking #1 above every 5★ competitor. Root cause: `rating*20 + reviews` let raw review volume swamp rating once a business crossed ~150 reviews. Replaced with a weighted-rating formula (`util.js`'s `rankByRating`/`weightedRating`, same idea as IMDb's ranking) where review count is a confidence weight on rating, not a separate additive score. Verified against the exact reported data (3.3★/176-review business correctly dropped to last) and against a live report (5★ businesses now correctly rank above 4.8★ regardless of review count). Shared between `places.js` and `mock.js` so both live and mock leaderboards use the same logic.
- **Lead notification cache-hit bug fixed.** `generate.js` returned early on a cache hit without ever firing the Formspree lead alert — a repeat request for an already-cached business was a live lead silently dropped. Fixed to fire on both paths, with the subject line distinguishing "new" vs "existing report requested".
- **Yelp removed entirely, replaced with Facebook + Nextdoor + X.** User decided against even the manual-entry compromise — dropped `api/_lib/yelp.js`, the admin UI's Yelp checkbox/fields, and all `yelpManual` plumbing in `pipeline.js`/`admin-generate.js`. Directory Presence is now Google (via Places) + Facebook + Nextdoor + X, the latter three all real and free via the same self-linked-page technique already proven for Facebook (`detectXLink`, `detectNextdoorLink` in `htmlscan.js`) — no signup needed for any of them. `directory.js`'s scoring formula had a hardcoded `/3` (platform count) that needed to become `/dirs.length` now that the list can vary. Also added a **Blog / content section** check to Business Details (`detectBlog`), same free self-scan technique, rescaled that category's score to include it. Verified: unit tests on the new detectors (X correctly ignores intent/share/widget links; Nextdoor and Blog match real patterns), a live report showing all 4 directory rows + the blog check with correct values, and the summary text's platform count is no longer hardcoded.

## Scoring restructure — "diagnosis, not penalties" (fixes mid-50s clustering)
- **Problem the user hit:** running many real businesses, they all scored in the 50s — even ones that should be near 100. Root cause was **not** bad data; it was the scoring math. Every category was forced to total 100 by handing out fixed points, so *not* having a chat widget, paid Google Ads tags, or an X/Nextdoor profile silently dragged scores down — punishing ordinary phone-first trades businesses for things they have no reason to do. Directory Presence capped a Google-only business at ~25; Techno Stack capped a no-paid-ads business at 60; Speed-to-Lead and Business Details capped phone-first sites in the 40s. Weighted together → everything lands mid-50s.
- **Fix — maturity-band scoring.** New `util.scoreChecks(checks)`: checks are either **core** (fundamentals — set the base as % of core weight passed) or **`bonus: true`** (nice-to-haves — passing *adds* credit, failing does nothing). So nailing the fundamentals reaches Excellent on its own, and extras can only lift a score, never create a penalty. Bands: Excellent 85+ / Good 65–84 / Needs Improvement 45–64 / High Priority <45 (`config.grade`).
- **Categories restructured** (8, per the user's product-direction doc, framed as "what would a customer's experience be, and where would they lose me"):
  Reputation (1.5) · Visibility (1.5, = GBP + site fundamentals HTTPS/mobile/indexable + "measures results" bonus) · Customer Experience (1.2, replaces Business Details) · Lead Capture (1.3, signature — **banded by how many contact channels exist**, no-chat ≠ failure) · Trust Signals (1.0, NEW — license/insured/testimonials/guarantee/certs/financing/BBB) · Content Quality (1.0, NEW) · Website Performance (1.0) · Competitive Position (1.2).
  **Dropped:** Directory Presence (X/Nextdoor penalty) and standalone Tech Stack (paid-ads penalty). Analytics presence survives as a light bonus check inside Visibility.
- **Content Quality — homepage read, free.** `api/_lib/content.js`: uses **Google Gemini free tier** (`GEMINI_API_KEY`, AI Studio) to judge clarity / CTA / trust / "would you hire them" + a one-line biggest-fix, rendered as a callout. Falls back to a **free pattern-based heuristic** when there's no key / call fails / rate limit hit — same "fall back to free" contract as the rest of the pipeline. Zero-cost by default; Gemini is opt-in. Note: free-tier inputs may be used by Google to improve models; only the business's public homepage text is sent.
- **New/changed files:** `util.js` (+`scoreChecks`), `config.js` (rewritten categories/bands/framing), `mock.js` (8 new builders), `htmlscan.js` (rewritten — contact-channel / trust-signal / site-basics detectors + new builders), `places.js` (`buildReputation` w/ review freshness, `buildVisibility` merging GBP+site, dropped `buildGbp`/`buildDirectoryPatch`), `leadcapture.js` (NEW — channel-count banding + phone-type), `content.js` (NEW — Gemini+heuristic), `pipeline.js` (rewired), `report-template.js` (labels/loading steps/content callout). **Deleted:** `directory.js`, `speedtolead.js`.
- **Verified:** mock + full pipeline produce a real spread (e.g. 14/73/100/90/54/100/79/25) instead of everything at ~50; live HTML scan against real trades sites (rotorooter, mrrooter) returns varied sane scores (30–100) with no crashes. **Known limitation:** heavily JS-rendered (SPA) sites return sparse initial HTML from a no-JS static fetch, so channel/content detection can undercount there — consistent with the "what we can read from your homepage" disclaimer.

## SPA / JavaScript-rendered site fix (three-agent research → free layered solution)
- **Problem:** the homepage scan does a plain `fetch()`, which returns raw HTML *before* JavaScript runs. SPA sites (many Wix/Squarespace/React builds) render content client-side, so the scan saw an empty shell and unfairly scored those businesses as "missing" contact info, trust signals, and content.
- **Approach chosen (via 3 parallel research agents):** reuse tools that already render the page rather than adding a headless browser. Rejected: headless Chromium (heavy bundle, cold starts, risks the 60s `vercel.json` maxDuration), "free" scraper trials (not actually free), and Gemini `url_context` (reads static HTML only — no help for SPAs).
- **Critical bug the research surfaced:** `gemini-1.5-flash` (the Content Quality default) was **deprecated Sept 2025 and no longer runs** — would have failed the instant a key was added. Fixed default to `gemini-2.5-flash` (`content.js`, `.env.example`).
- **What shipped (all free, no new deps):**
  1. **`api/_lib/structured.js` (NEW)** — parses schema.org JSON-LD + OpenGraph/meta from the already-fetched HTML. Wix/Squarespace/Duda inject this server-side even on SPAs, so it recovers phone / email / hours / address / social links / star rating the regex misses. Threaded into `detectChannels`/`detectHours`/`detectTrust` and the builders.
  2. **PageSpeed reuse (`pagespeed.js`)** — the mobile Lighthouse run now also requests `category=seo`; new `parseRendered()` extracts the *rendered* (post-JS) `viewport`/`is-crawlable` audits and the full-page **screenshot** (`fullPageScreenshot.screenshot.data`). `runPerformance` now returns `{ category, rendered }`. `detectSiteBasics` prefers these rendered signals over regex (correct for SPAs).
  3. **Gemini vision (`content.js`)** — on a thin shell (or when the fetch failed) it sends the PageSpeed-rendered screenshot to Gemini as `inline_data` and reads the homepage as a customer sees it; falls back to Gemini text, then the free heuristic.
  4. **Thin-shell guard** — `util.scoreChecks` gains a third check kind, `neutral: true` (excluded from base AND bonus — display-only). `htmlscan.isThinShell()` flags near-empty JS shells; builders mark unrecoverable checks neutral ("Unknown (JavaScript site)", rendered as a muted "–" via `report-template.checkRow` + `.chk-unknown` in `checkup.css`, `?v=2`). When too little is recoverable to score honestly, builders return `null` so the pipeline keeps the neutral mock baseline instead of emitting an unfair 0 or an inflated 100.
- **Shared context object** `ctx = { structured, rendered, thinShell }` threaded from `pipeline.js` into every HTML builder; `siteSignalsForVisibility` now runs when EITHER the HTML fetch OR the PageSpeed render succeeded (heavy SPAs often fail the fetch but render fine in PageSpeed), so Visibility keeps its site half either way. No new network calls in the concurrent phase; the vision call replaces (not adds to) the text Gemini call.
- **Verified:** structured parser recovers phone/email/hours/rating/social from a JSON-LD SPA shell; a truly empty shell returns null (keeps mock) rather than failing; neutral checks render as muted "–"; full pipeline + all modules load and run clean. **Note:** the free-tier Gemini data policy (inputs may train Google's models) applies — only public homepage text/screenshots are sent. Free-tier limit ~250 reports/day on 2.5-flash.
